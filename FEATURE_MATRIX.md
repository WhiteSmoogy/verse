# Verse Feature Matrix

This project is moving toward Verse language compatibility in measured batches.
Every implemented feature listed here has at least one regression test in
`tests/language.rs`.

## Architecture layers

Features are split into two layers by how tightly they bind to a host runtime:

- **Core language** — everything that can be fully evaluated in a plain process
  with no Fortnite/UEFN runtime: lexing, types, failure contexts, effects,
  modules, classes/interfaces, containers, and the free-standing standard library
  (math, `color`/`NamedColors`, `message`/`Localize`, string helpers). These are
  the bulk of the tables below and are the project's primary target.
- **Platform layer (UEFN / Fortnite runtime)** — symbols that only have meaning
  given a running match: simulation world objects (`player`/`agent`/`team`/
  `entity`), the `session` lifecycle, and player-scoped persistence
  (`weak_map(player, …)`). These are checked statically but cannot be fully
  evaluated without a Fortnite runtime, so they are collected in their own
  section ([Platform Layer](#platform-layer-uefn--fortnite-runtime)) and kept
  optional rather than treated as core-language debt.

The dividing line: *if it can be completely evaluated without a match/player/
asset/scheduler, it is core; otherwise it is platform.* By that rule `color` is
core (a pure value plus math), while `GetSession()` is platform.
